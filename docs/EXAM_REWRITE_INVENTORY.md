# Exam Rewrite Phase 0 Inventory

This inventory is the required Phase 0 map for `docs/EXAM_REWRITE_PLAN.md`. It records the current assessment system before any rewrite work changes behavior.

## Conservative Assumptions

- Integrity controls are deterrents only. They may block copy/paste, detect focus/fullscreen/devtools signals, warn the learner, auto-submit, or zero a score through the existing violation path, but they are not treated as real proctoring.
- Exam and quiz authoring support the existing item kinds only: `CHOICE` and `MATCHING`. `OPEN_TEXT`, `FORM`, and `CODE` remain available to other assessment surfaces only where already supported.
- The existing course shell, student activity route, grading engine, and analytics pages stay in place. The rewrite must wrap and converge them, not replace them.
- Compatibility aliases in legacy settings are migration inputs only. New frontend state must read canonical DTOs from the API once Phase 1 lands.
- Any item bank, bulk import, AI generation, or new question type work is out of scope until after the rewrite acceptance criteria are satisfied.

## Data Flow

1. Teacher creates an exam in `ExamActivityModal`. The modal sends a settings object with legacy keys such as `time_limit`, `attempt_limit`, `shuffle_questions`, `allow_result_review`, `show_correct_answers`, and integrity booleans.
2. Backend creates or reads canonical rows: `Activity`, `Assessment`, `AssessmentPolicy`, `AssessmentItem`, and access policy rows.
3. Studio reads `GET /assessments/activity/{activity_uuid}`. The response contains `assessment_policy`, `items`, `attempt_projection`, `review_projection`, `content_version`, and `policy_version`.
4. Current studio state is reconstructed in `studio/utils.ts` from `assessment_policy`, `assessment_policy.settings_json`, and `assessment_policy.anti_cheat_json`. This is where most aliasing and duplicate derivation exists.
5. Studio saves through `PATCH /assessments/{assessment_uuid}` with `policy.due_at`, `policy.max_attempts`, `policy.time_limit_seconds`, `policy.anti_cheat_json`, and `policy.settings_json`.
6. Student runtime reads `useAssessmentAttempt`, which maps `assessment.attempt_projection.effective_policy` into `PolicyView`. Attempt buttons, due state, timer, and release state are mostly server-driven.
7. Server attempt enforcement runs through grading pipeline helpers:
   - `enforce_attempt_limit`
   - `enforce_time_limit`
   - `enforce_late_submission`
   - `check_violations`
8. Submission snapshots preserve item and policy data at submit time.

## Student Runtime Policy Behavior

| Policy field                  | Runtime behavior verified                                                                                                                             |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| Lifecycle/publish state       | Attempt projection blocks drafts, scheduled items before `scheduled_at`, and archived assessments for learners.                                       |
| Attempts                      | Attempt projection computes attempts remaining; grading pipeline enforces completed attempt count. Teachers preview as unlimited.                     |
| Due date and late submissions | Attempt projection blocks past due when late submissions are disabled; pipeline enforces late submissions and late penalties.                         |
| Time limit                    | Attempt projection calculates server-authoritative `timer_expires_at`; guard auto-submits on expiry; pipeline enforces with 30 second tolerance.      |
| Grade release                 | Student result visibility is driven by published grades/release state, not only by `allow_result_review`.                                             |
| Integrity                     | Frontend guard detects configured signals and sends violation count on submit; pipeline can zero score if configured.                                 |
| Access                        | Course access plus assessment access policy restrict submit eligibility.                                                                              |
| Per-student overrides         | Max attempts, due date, and late penalty waiver exist. Time-limit override appears in DTOs but is not applied in the inspected effective policy path. |
